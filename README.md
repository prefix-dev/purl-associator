# PURL Associator

A workbench for associating [Package URLs (PURLs)](https://github.com/package-url/purl-spec)
with conda-forge packages, so security tools (OSV, AboutCode, etc.) can
correlate CVEs against conda artifacts.

The repository is two things at once:

1. **The mapping data itself**, kept in `mappings/`:
   - `auto.json` — generated by a Python script that walks conda-forge
     metadata and infers PURLs from each package's rendered recipe.
   - `manual.json` — human-curated overrides committed via PR.
2. **A small site that edits that data.** A static React/Vite app
   (`web/`) reads the JSON, lets a logged-in user batch up edits, and pushes
   them as a pull request against this repo via the GitHub API. Auth uses a
   tiny Cloudflare Worker (`worker/`) for the OAuth code-exchange step.

## Components

```
.
├── mappings/              ← source of truth (JSON, edited via PRs)
│   ├── auto.json
│   └── manual.json
├── scripts/               ← Python automapper (py-rattler)
│   ├── automap.py
│   ├── purl_inference.py
│   └── merge_mappings.py
├── web/                   ← React/Vite SPA (deployed to GitHub Pages)
├── worker/                ← Cloudflare Worker for GitHub OAuth code-exchange
├── pixi.toml
└── .github/workflows/
    ├── automap.yml        ← cron + workflow_dispatch (refresh auto.json)
    ├── pages.yml          ← deploy SPA to GitHub Pages
    └── worker.yml         ← deploy worker to Cloudflare
```

## Local development

```sh
# 1. Generate or refresh the auto mapping (uses py-rattler).
pixi run python -m scripts.automap --only numpy,ripgrep,pandas
pixi run python -m scripts.merge_mappings   # writes web/public/mappings.json

# 2. Run the SPA in dev mode.
cd web && npm install && npm run dev
```

The dev SPA reads `./mappings.json` from its own origin. With no OAuth
configured the site is read-only; "Sign in" returns an explanation.

## Auto-mapping in detail

`scripts/automap.py`:

1. Uses `rattler.repo_data.Gateway` to enumerate every package name in the
   `conda-forge` channel across `linux-64` + `noarch`.
2. For each name, picks the most-recent record by `(version, timestamp,
   build)`.
3. Fetches `info/recipe/rendered_recipe.yaml` (rattler-build) or
   `info/recipe/meta.yaml` (conda-build) via
   `rattler.package_streaming.fetch_raw_package_file_from_url` — a partial
   download that pulls just the requested entry from the package archive.
4. Extracts source URLs and host/build dependency lists; runs
   `scripts.purl_inference` heuristics:
   - `pypi.org` / `files.pythonhosted.org` → `pkg:pypi/<name>`
   - `github.com/<owner>/<repo>` → `pkg:github/<owner>/<repo>`
   - `crates.io` → `pkg:cargo/<name>`
   - `registry.npmjs.org` → `pkg:npm/[<scope>/]<name>`
   - `cran.r-project.org` → `pkg:cran/<name>`
   - `bioconductor.org` → `pkg:bioconductor/<name>`
   - else `pkg:generic/<name>`
   - **Recipe context** wins ties: a Python build (pip / meson-python /
     maturin / cython, or `noarch: python`) prefers `pkg:pypi/...` over a
     `pkg:github/...` source URL, so e.g. numpy maps to `pkg:pypi/numpy`
     even though its tarball is on github releases.
5. Writes the result to `mappings/auto.json`. The script is incremental —
   re-runs skip packages whose `(version, build)` is unchanged from the
   prior file.

The default `pixi run automap` runs against the entire channel (~25k
packages); use `--only numpy,ripgrep,…` or `--limit 200` while iterating.

## Editing through the SPA

The app is read-only by default. When the user clicks any field that would
edit a mapping, a "Sign in with GitHub" modal appears. After OAuth:

1. The frontend reads/forks `prefix-dev/purl-associator` (or whatever this
   repo is configured as).
2. Creates a branch `purl-mapping/<user>-<short>`.
3. PUTs an updated `mappings/manual.json` via the contents API.
4. Opens a PR back to `main` titled with the user-supplied message.

All of this happens client-side against `api.github.com`, which does
support CORS for authenticated requests.

## Deployment

### GitHub Pages

Set repo variables (Settings → Secrets and variables → Actions → Variables):

| Variable                  | Value                                              |
| ------------------------- | -------------------------------------------------- |
| `OAUTH_WORKER_URL`        | `https://purl-associator-oauth.<account>.workers.dev` |
| `GITHUB_OAUTH_CLIENT_ID`  | client ID of your GitHub OAuth App                  |

Push to `main`; the `pages` workflow builds and publishes the SPA. Set the
GitHub OAuth App's "Authorization callback URL" to the Pages URL.

### Cloudflare Worker

```sh
cd worker
npm install
wrangler login
wrangler secret put GITHUB_CLIENT_SECRET   # from your GitHub OAuth App
# Update wrangler.toml's ALLOWED_ORIGINS to include your Pages URL.
wrangler deploy
```

The worker exposes `POST /exchange` which trades an OAuth `code` for an
`access_token`. The frontend then uses that token directly with
`api.github.com`.

For automated deploys via GitHub Actions, set repo secrets
`CLOUDFLARE_API_TOKEN` and `CLOUDFLARE_ACCOUNT_ID`; the `worker.yml`
workflow handles the rest.

### Scheduled automap

`automap.yml` runs every 24 hours (cron `17 3 * * *` UTC) and on manual
dispatch. It:

1. Re-runs the Python automapper.
2. Merges with `manual.json` into `web/public/mappings.json`.
3. Opens a PR titled `automap: refresh PURL mappings` if anything changed
   (uses `peter-evans/create-pull-request`). Reviewing this PR is the gate
   on what gets published.
